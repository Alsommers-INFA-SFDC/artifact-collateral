1. Data Domains: The Menu360 Business Entities

▼

A restaurant franchise system manages multiple master data domains that work together to create a complete operational picture. Each domain represents a distinct business entity with its own lifecycle, governance rules, and relationships to other domains.

🏢

Organization

Represents the franchise group or parent company that operates multiple restaurant locations. Organizations manage brand portfolios, pricing strategies, and governance rights.

Key Attributes:
  • • Legal entity name and tax ID
  • • Brands operated (multi-brand franchisees)
  • • Pricing autonomy level and markup caps
  • • Rights to add local menu items
🏬

Location

Represents individual physical restaurant locations. Each location subscribes to corporate menu catalogs but can apply local pricing overrides and opt-outs based on franchise agreements.

Key Attributes:
  • • Store number and physical address
  • • Parent organization reference
  • • Menu subscriptions and overrides
  • • Location features (drive-thru, delivery, etc.)
📖

Menu Catalog

Groups recipes into organized categories with base pricing (MSRP). Supports multiple menu types including core, seasonal (LTO), and channel-specific menus for delivery platforms.

Key Attributes:
  • • Menu name, type, and status
  • • Categories with display order
  • • Base MSRP for each recipe
  • • Daypart availability and date ranges
🍔

Recipe

The core product entity representing any sellable item. Contains formulations, ingredient lists, nutritional information, and syndication profiles for various downstream systems.

Key Attributes:
  • • Name, description, and item type
  • • Ingredient formulation and yields
  • • Syndication aliases (POS, OLO, menuboard)
  • • Linked modifier groups for customization
🥓

Ingredient

Raw or prepared components used in recipes. Tracks source ingredients for allergen tracing, nutritional specs, and preparation rules like cook temperature or style preferences.

Key Attributes:
  • • Ingredient name and type
  • • Source ingredients list (for allergens)
  • • Nutritional information (calories, etc.)
  • • Preparation rules and cooking options
➕

Modifier Group

Manages customization options for recipes. Defines selection rules (min/max), UI presentation (radio/checkbox), price deltas, and available options for customer personalization.

Key Attributes:
  • • Group name and description
  • • Selection rules (min/max/free count)
  • • UI presentation type
  • • Modifier options with price deltas

💡 Key Concepts: Relational Master Data & Customization

  • • Recipe is the parent record that defines the sellable product (Classic Cheeseburger).
  • • Each ingredient is a separate master record with full traceability (allergens, nutrition, sourcing).
  • • Modifier groups enable customization - Each modifier group links the recipe to multiple ingredient options (e.g., choose your bun, choose your patty).
  • • Same ingredients, different contexts - Chicken Breast (ING-301) appears in both "Core" and "Game Day" modifier groups, showing ingredient reuse across different menu contexts.
  • • Relationships are maintained via MDM IDs - Recipe → Modifier Groups → Ingredient IDs, creating a full customization matrix.
  • • Updates cascade automatically - Change an ingredient's allergens once (e.g., ING-301), it updates everywhere that ingredient is used.

6. Menu Syndication Example

▼

The core benefit of using a Menu repository with Context (Menu360) is the ability to manage all of these disparate data points and build the context between them. Instead of managing in disconnected Excel sheets, you manage individual entities and relate them together for accurate and governed information.

This new context-aware menu repo is now syndicated to Online Ordering and POS systems, regardless of how they receive the data. For example, while DoorDash, Grubhub, and Uber Eats all ingest the same core menu items and food customizations from an upstream PIM system like Informatica P360, their API architectures organize this data in fundamentally different ways:

1 Structural Design Models

  • DoorDash (Hierarchically Nested Tree): Data follows a strict parent-to-child lineage embedded directly inside a single object cascade. Drill deep to find modifier choices.
  • Grubhub (Normalized Sibling Arrays): Data is entirely flattened at the root level into discrete, independent tables, linked via string reference IDs (e.g., "items": ["ITEM_ID"]).
  • Uber Eats (Relational Entity Graph): Treats menus, categories, items, and modifier groups as modular, globally unique entities explicitly cross-referenced at each level.

2 Pricing & Float Notation

  • DoorDash & Uber Eats: Integer Cents Only. A price of $12.99 must be exactly 1299. Decimal floats are rejected.
  • Grubhub: Uses standard Decimal Floats (e.g., 12.99).

3 Text & Localization Handling

  • DoorDash & Grubhub: Accept direct, flat text strings (e.g., "name": "Cheeseburger").
  • Uber Eats: Mandates a strict localization map structure. Text must be isolated inside a nested translations block (e.g., {"translations": {"en_us": "..."}}).

4 Selection Constraint Logic

  • DoorDash: Governed by min_selection and max_selection nested within the modifier group.
  • Grubhub: Handled via decoupled reference identifiers called modifier_prompts.
  • Uber Eats: Governed by a quantity_info block using min_permitted and max_permitted variables.
