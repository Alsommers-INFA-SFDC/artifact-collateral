Overview

In today's digital food service landscape, Quick Service Restaurants (QSRs) must manage complex menu data across multiple delivery platforms including DoorDash, Grubhub, and Uber Eats. Each platform requires different data formats, structures, and standards for menu items, customizations, pricing, and availability. Informatica Master Data Management (MDM) Hub provides a centralized solution to manage this complexity, enabling consistent, accurate, and efficient menu data delivery across all channels.

Key Challenge

While DoorDash, Grubhub, and Uber Eats all ingest the same core menu items and food customizations from an upstream PIM system like Informatica P360, their API architectures organize this data in fundamentally different ways.

API Architecture Differences

Understanding the fundamental architectural differences between delivery platforms is critical for successful data transformation. Each platform has evolved unique data structures, pricing models, and localization requirements that must be precisely accommodated during the data export process from Informatica MDM Hub.

1. Structural Design Models

DoorDash: Hierarchically Nested Tree

Data follows a strict parent-to-child lineage embedded directly inside a single object cascade. To find a modifier choice, the processing engine must drill deep into the specific menu, category, and item array where it resides. This deeply nested structure requires recursive traversal logic when building data exports from P360.

{ "menu": { "categories": [{ "items": [{ "modifier_groups": [{ "modifiers": [...] }] }] }] } }

Grubhub: Normalized Sibling Arrays

Data is entirely flattened at the root level into discrete, independent tables. Entities are linked exclusively via a web of string reference IDs ("items": ["ITEM_ID"]). No item or modifier data is ever nested inside another. This approach requires careful relationship mapping and ID management during export from P360.

{ "menus": [{"id": "MENU_1", "category_ids": ["CAT_1"]}], "categories": [{"id": "CAT_1", "item_ids": ["ITEM_1"]}], "items": [{"id": "ITEM_1", "modifier_group_ids": ["MOD_1"]}], "modifier_groups": [{"id": "MOD_1"}] }

Uber Eats: Relational Entity Graph

Data treats menus, categories, items, and modifier groups as modular, globally unique entities mapped inside their own explicit root arrays. Cross-references are explicitly declared at each level to map relationships. This hybrid approach combines elements of both nested and flat structures, requiring sophisticated entity resolution during P360 export.

{ "menus": [{"id": "menu-uuid", "category_ids": [...]}], "categories": [{"id": "cat-uuid", "entities": [{"type": "ITEM"}]}], "items": [{"id": "item-uuid", "modifier_group_ids": {...}}] }

2. Pricing and Float Notation

DoorDash

Integer Cents Only
A price of $12.99 must be multiplied by 100 to map as an exact integer value (1299). Decimal floats are rejected by the API.

"price": 1299

Grubhub

Decimal Floats
Prices are sent natively as standard currency values (e.g., 10.00 or 1.50). No multiplication required.

"price": 12.99

Uber Eats

Integer Cents Only
Same as DoorDash. Price of $12.99 must be sent as integer 1299. Requires currency conversion logic in P360 export.

"price": 1299

3. Text and Localization Handling

DoorDash

Direct Flat Strings

"name": "Classic Cheeseburger"

Grubhub

Direct Flat Strings

"name": "Classic Cheeseburger"

Uber Eats

Mandatory Localization Map

"title": { "translations": { "en_us": "Classic Cheeseburger" } }

Uber Eats Localization Requirement

Every single text field must be isolated inside a nested translations block mapped to an ISO language code. Plain text inputs fail validation. This requires P360 to wrap all consumer-facing text in translation objects during export.

4. Selection Constraint Logic (Min / Max Choices)

DoorDash

Governed by min_selection and max_selection integers nested within the modifier group schema.

{ "min_selection": 1, "max_selection": 3 }

Grubhub

Does not handle complex selections in the core item payload directly; uses standalone, decoupled reference identifiers called modifier_prompts.

{ "modifier_prompt_id": "PROMPT_1" }

Uber Eats

Governed by a quantity_info nested block using min_permitted and max_permitted variables.

{ "quantity_info": { "min_permitted": 1, "max_permitted": 3 } }

P360 Integration Challenge

These architectural differences require Informatica P360 (Product 360) to maintain three distinct export templates, each with custom transformation logic for structure (nesting vs. flattening), pricing (integer vs. decimal), localization (plain text vs. translation objects), and relationship mapping (embedded vs. ID references). The MDM Hub orchestrates these transformations to ensure accurate, platform-specific data delivery.
