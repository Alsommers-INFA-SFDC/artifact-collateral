Primary Focus B2C

  • Customer ID (Master), Cross-References (Source System IDs)

  • Loyalty (Tier, Points, Membership Date)
  • Preferences (Channel, Communication, Product Categories)
  • Transactional (CLV, Last Purchase Date, Total Spend, Frequency)

Configuration:

  • Trust Model: Consolidated trust (multiple sources, single golden record)
  • Matching: Probabilistic + deterministic rules (email, name, phone, address)
  • Survivorship: Source priority + recency + completeness
  • Workflow: Auto-merge above 95 score, manual review 70-94

Domain Integration Architecture

Data Flow & Synchronization

  • Inbound: Source systems feed domain data to MDM via APIs and batch loads
  • MDM Processing: Match, merge, data quality, survivorship applied per domain
  • Cross-Reference: Source system IDs linked to golden records (XREF tables)
  • Outbound: Golden records published to consuming systems (CRM, E-commerce, ERP, BI)
  • Event-Driven: CDC events trigger downstream updates in real-time
  • Bi-Directional: Some domains support write-back (updates from consuming systems flow back to MDM)

Data Quality

Data Quality Framework

Comprehensive data quality processes ensure clean, standardized, and enriched customer records:

Validation Rules

Format Validation

  • Email: RFC 5322 format validation, disposable domain check
  • Phone: E.164 format, country code validation
  • Postal Code: Country-specific format rules
  • Date of Birth: Valid date, age range check (13-120 years)

Business Rule Validation

  • Required Fields: First Name, Last Name, Email OR Phone
  • Unique Constraints: Email must be unique per customer
  • Referential Integrity: Loyalty ID must exist in loyalty system
  • Value Domains: Gender, Country, State from approved lists

Standardization & Cleansing

Address Standardization

  • Vendor: Informatica Address Verification (USPS, international databases)
  • Process: Parse → Validate → Standardize → Geocode
  • Output: Corrected address, standardized format, delivery point barcode, latitude/longitude
  • Quality Scores: Address Quality Index (AQI) score, delivery confidence level

Name Parsing & Standardization

  • Parse full name into components: Prefix, First, Middle, Last, Suffix
  • Case normalization: Proper case (Smith not SMITH)
  • Special character handling: Preserve accents, hyphens, apostrophes
  • Nickname expansion: Bob → Robert (stored in alternate name field)

Email & Phone Cleansing

  • Email: Lowercase, trim whitespace, check MX records, flag disposable domains
  • Phone: Remove formatting, normalize to E.164 (+1XXXXXXXXXX), validate area code

Enrichment

Data Enrichment Services

  • Demographic Append: Age, income range, household size from third-party providers
  • Lifestyle Segmentation: Interest categories, purchase propensity scores
  • Geocoding: Latitude/longitude from address
  • Company Enrichment: For B2B, append company size, industry, revenue

Data Quality Scoring

Completeness Score

Calculated based on presence of key attributes (0-100):

  • Name fields populated: 20 points
  • Valid email: 20 points
  • Valid phone: 15 points
  • Complete address: 25 points
  • Date of birth: 10 points
  • Preferences captured: 10 points

Accuracy Score

Based on validation results (0-100):

  • Address verified (AQI score)
  • Email verified (MX check)
  • Phone validated (carrier lookup)
  • Recent data update (within 90 days)

Data Quality Monitoring

Quality Dashboards

  • Record Quality Distribution: Histogram of completeness/accuracy scores
  • Trend Analysis: Quality score trends over time
  • Source System Comparison: Quality by source system
  • Issue Tracking: Count of validation failures by rule

Match & Merge

Match & Merge Strategy

Multi-pass matching algorithm to identify and consolidate duplicate customer records:

Match Rules Configuration

Match Rule Set: Customer Exact Match

Priority 1 Deterministic

  • Criteria: Email = Email (exact match)
  • Score: 100 (auto-match)
  • Action: Auto-merge
  • Use Case: Same person registers twice with same email

Match Rule Set: High Confidence Fuzzy Match

Priority 2 Probabilistic

  • Criteria:
    • First Name (fuzzy, 85% similarity) AND Last Name (fuzzy, 90% similarity)
    • AND (Phone = Phone OR Address = Address)
  • Score: 85-95 (manual review)
  • Action: Flag for steward review
  • Use Case: Typos in name, same person different channel

Match Rule Set: Household Match

Priority 3 Probabilistic

  • Criteria:
    • Different First Name, Same Last Name
    • AND Address = Address (standardized)
  • Score: 70-80
  • Action: Link as household members (do not merge)
  • Use Case: Family members at same address

Match Rule Set: Low Confidence Match

Priority 4 Probabilistic

  • Criteria: Name (fuzzy 70%) AND Date of Birth AND Partial Phone Match
  • Score: 60-70
  • Action: Flag for review, do not auto-merge
  • Use Case: Possible match but insufficient confidence

Matching Algorithms

Fuzzy Matching Techniques

  • Jaro-Winkler: String similarity for names (weights prefix matches higher)
  • Levenshtein Distance: Edit distance for email, account numbers
  • Soundex/Metaphone: Phonetic matching for names (Smith = Smyth)
  • Token-Based: Break address into tokens, match on components
  • N-Gram: Character sequence matching for misspellings

Match Workflow

Match Process Flow

  1. Trigger: New record loaded or existing record updated
  2. Standardization: Apply data quality rules (name case, address standardization)
  3. Blocking: Generate match keys (e.g., first 3 chars of last name + postal code) to reduce search space
  4. Candidate Selection: Find records with matching block keys
  5. Match Rule Execution: Apply match rules in priority order
  6. Scoring: Calculate match score (0-100)
  7. Auto-Merge: Score ≥ 95 → auto-merge
  8. Manual Review: Score 70-94 → steward task
  9. No Match: Score < 70 → create new master record

Merge Process

Merge Execution Steps

  1. Identify Records: Parent records to merge (based on match score or manual selection)
  2. Apply Survivorship: Determine winning value for each field (see Survivorship section)
  3. Create Golden Record: New master record with survived values
  4. Update Cross-References: Link all source system IDs to golden record
  5. Archive Parent Records: Move original records to merge history
  6. Propagate Changes: Publish merge event to downstream systems

Unmerge Process

Support for splitting incorrectly merged records:

  • Trigger: Steward identifies incorrect merge or customer dispute
  • Process: Restore original parent records, remove golden record, update XREFs
  • History: Log unmerge reason, steward ID, timestamp
  • Downstream Impact: Publish unmerge event to notify consuming systems

Survivorship

Survivorship Strategy

Rules to determine which field values "survive" to the golden record when merging duplicate customer records:

Survivorship Rule Types

Source Priority

Most Common

Trust hierarchy based on source system reliability:

  1. CRM (customer self-update) - Highest trust
  2. E-commerce (user registration)
  3. POS (in-store data)
  4. Third-party enrichment - Lowest trust

Applied to: Email, phone, address, preferences

Recency

Most Current

Most recently updated value wins

Applied to: Phone number, address, communication preferences, loyalty tier

Exception: If recent value is NULL, keep older non-NULL value

Completeness

Most Complete

Value with most complete data wins

Applied to: Full name (prefer records with middle name populated), address (prefer records with unit/apt number)

Example: "John M. Smith" beats "John Smith"

Longest Value

Length-Based

Value with most characters wins

Applied to: Company name, product descriptions

Rationale: Longer values typically more descriptive

Field-Level Survivorship Configuration

Customer Domain Survivorship Rules

Field Survivorship Rule Rationale
First Name Source Priority → Completeness Trust CRM data, prefer full name over nickname
Last Name Source Priority → Recency Trust CRM, handle name changes (marriage)
Email Source Priority → Recency → Data Quality Score Trust self-reported, prefer verified emails
Phone Recency → Source Priority Phone numbers change frequently
Address Recency → Data Quality Score Prefer validated addresses, recent moves
Date of Birth Source Priority → Most Complete DOB doesn't change, prefer full date over partial
Gender Source Priority (CRM self-reported) Trust self-identification
Loyalty Tier Source Priority (Loyalty System) Single source of truth
Customer Lifetime Value Max Value Take highest calculated CLV
Marketing Opt-In Recency + Legal Flag Most recent consent, opt-out always wins

Advanced Survivorship Patterns

Conditional Survivorship

Rules that apply different logic based on conditions:

  • Opt-Out Override: If ANY source says opt-out = TRUE, golden record opt-out = TRUE (most restrictive wins)
  • NULL Value Handling: If recent value is NULL, fallback to older non-NULL value (don't lose data)
  • Data Quality Threshold: If recent value has quality score < 50, use older value with higher score
  • Business Hours Update: For phone, prefer business hours (9-5) sources over batch loads

Multi-Value Survivorship

For fields that allow multiple values (e.g., addresses, phone numbers):

  • Merge All: Retain all unique values from all parent records
  • Deduplicate: Remove duplicate values (after standardization)
  • Primary Flag: Apply survivorship to determine which value is "primary"
  • Active Flag: Mark recent values as active, old values as inactive (don't delete)

Survivorship Transparency

Audit & Lineage

Every field value in the golden record includes metadata:

  • Source System: Which source provided this value
  • Source Record ID: Original record identifier
  • Load Date: When value entered MDM
  • Update Date: When value last changed
  • Survivorship Rule: Which rule determined this value won
  • Data Quality Score: Quality score at time of load
